# Anti-Patterns — アンチパターン集と修正

実例から学ぶ「やりがちな設計ミス」。見つけたら直す。

## 1. 1 列に複数値 (Jaywalking)
区切り文字・配列・エンコード文字列で複数値を 1 セルに詰める。
- **問題**: 個別検索・結合・集計が不能、整合性が保てない、索引が効かない。
- **修正**: 子テーブル、または多対多なら中間（junction）テーブル。
```sql
-- ❌  tags TEXT  -- "rock,jazz,blues"
-- ⭕
CREATE TABLE song_tags (
  song_id INTEGER NOT NULL REFERENCES songs(id) ON DELETE CASCADE,
  tag     TEXT    NOT NULL,
  PRIMARY KEY (song_id, tag)
);
```

## 2. 主キーなし
複合自然キー（氏名＋生年月日 等）で代用、または PK 無し。
- **問題**: 重複行、行を一意に指せず更新/削除不能。医療記録で患者取り違えの実例あり。
- **修正**: サロゲート主キーを必ず置く。自然キーは別途 UNIQUE。

## 3. FK 制約を貼らない（ID だけ持つ）
参照 ID を持つが FK 制約を宣言しない。
- **問題**: 孤児レコードが溜まる。「完璧なコード」を前提にした楽観。日々の検査が必要に。
- **修正**: エンジンが対応する限り FK 制約を宣言する（D1/SQLite は対応）。

## 4. ポリモーフィック関連
1 つの FK 列 + `type` 列で複数テーブルを参照（`commentable_id` + `commentable_type`）。
- **問題**: FK は複数表にまたがれず参照整合性を貼れない。孤児が溜まり、クエリが
  CASE 地獄、親削除がカスケードしない。
- **修正**: 関連ごとに別々の FK 列（nullable）を持つ、または交差テーブルを関連ごとに分ける。

## 5. 1 テーブルにエンティティ混在
別種のエンティティを `type` 列と大量の nullable 列で 1 表に詰める。
- **問題**: NULL 列だらけ、制約が効かない、意味が曖昧。
- **修正**: エンティティごとにテーブルを分け、明示的な関連で結ぶ。

## 6. EAV (Entity-Attribute-Value)
`(entity, attribute, value)` の何でも箱で可変属性を表現。
- **問題**: 型・制約・索引・集計がすべて弱くなる。クエリが激重。
- **修正**: 本当に固定スキーマにできないなら **JSON 列**に限定的に寄せる（検証はアプリ）。
  検索/集計が要る属性は通常列に昇格させる。

## 7. 無計画なソフトデリート
DELETE せず常に隠すだけ（フラグ運用）。
- **問題**: 参照整合性の保証を失い、カスケード削除をアプリで手動管理する羽目に。
  UNIQUE 制約が削除済み行と衝突する。
- **修正**: 本当に履歴が要るときだけ採用。`deleted_at TIMESTAMP NULL` + 部分索引
  `WHERE deleted_at IS NULL`、UNIQUE は部分 UNIQUE 索引にする。要らないなら素直に DELETE。

## 8. 浮動小数で金額
`REAL`/`FLOAT` で通貨を持つ。
- **問題**: 丸め誤差で 1 円が合わない。
- **修正**: 最小単位の整数（銭/cent）で持つ、または DECIMAL 相当。

## 9. EXPLAIN を見ずに索引追加
当てずっぽうで索引を貼る/貼らない。
- **問題**: 効かない索引で write だけ重くなる、または全走査に気づかない。
- **修正**: `EXPLAIN QUERY PLAN` で実際の利用を確認してから貼る/外す。`indexing.md` 参照。

## 10. 命名・型の不統一
表ごとに `created`/`createdAt`/`create_time` が混在、時刻が文字列だったり整数だったり。
- **問題**: 認知負荷・バグ・移行の困難。
- **修正**: 命名規約と型をプロジェクトで 1 つに固定（`schema-design.md` 参照）。
