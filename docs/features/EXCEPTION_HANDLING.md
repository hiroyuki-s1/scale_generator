# 例外処理 設計方針（全機能横断）

新機能（ログイン / ソングブック / 度数カラー / ポジション表示制御 / リリースノート）の
例外処理を一覧化する。**「落とさない・黙らせない・ユーザーに伝える」** が原則。

---

## 0. 共通原則

1. **ユーザーデータを失わない**: API 失敗時もローカル（localStorage）は保持。
   読み込み等の破壊的操作は確認ダイアログ＋成功確定まで旧データを保持。
2. **楽観的 UI + ロールバック**: 保存/削除は UI 先行、失敗時に元へ戻しトースト表示。
3. **黙ってswallowしない**: `catch` では必ず `console.error` にコンテキストを残す。
4. **ユーザー向け文言は平易に**: 技術的詳細（スタック等）は UI に出さない。
5. **多重実行防止**: 通信中ボタンは disabled / 二重送信ガード。

---

## 1. 認証（Clerk）

| ケース | 挙動 |
|--------|------|
| Clerk SDK のロード失敗（ネット断/ブロッカー） | ログインボタンを無効化し「ログイン機能を読み込めませんでした」。アプリ本体（ローカル機能）は通常動作 |
| ログインポップアップをユーザーが閉じた | 何もしない（エラー扱いにしない） |
| JWT 期限切れ（API が 401） | サイレントにトークン再取得 → 再試行。再取得も失敗ならログアウト状態にしてトースト「再ログインしてください」 |
| サードパーティ Cookie ブロック | ログイン不可の旨を案内。ローカル機能は継続 |
| パスワード再設定（メール未着 等） | Clerk の標準フローに委譲（再送はClerk UI）。アプリ側は介在しない |
| 退会 Webhook の D1 削除失敗 | Clerk は Webhook 失敗時に**自動リトライ**するので、各テーブル DELETE は冪等に（再送で 0 件でも 200）。一部テーブルのみ失敗時も再送で残りが消える |
| 退会後もローカルに残るデータ | localStorage は端末側。退会＝サーバ削除であり、端末のローカル利用は継続可（仕様どおり） |

## 2. ソングブック（D1 / Pages Functions）

### 2.1 ネットワーク／API レイヤー

| ケース | 挙動 |
|--------|------|
| 一覧取得失敗（GET 500/タイムアウト） | 「ソングブックを取得できませんでした。再読み込み」ボタンを表示 |
| 保存失敗（POST 失敗） | 楽観的に追加したカードを除去 → トースト「保存に失敗しました」 |
| 削除失敗（DELETE 失敗） | 楽観的に消したカードを復元 → トースト「削除に失敗しました」 |
| 読み込み失敗（GET :id 失敗） | **現在のソングファイルは保持**（展開前に失敗を確定）→ トースト「読み込みに失敗しました」 |
| 401（未認証/期限切れ） | §1 のトークン再取得フローへ |
| 403（他人の id にアクセス） | 「権限がありません」。一覧を再取得 |
| 429（レート制限） | 「混み合っています。しばらくして再試行」 |
| オフライン（`navigator.onLine === false`） | API を叩く前に「オフラインです」を表示し送信しない |

### 2.2 入力／バリデーション

| ケース | 挙動 |
|--------|------|
| ソングブック名が空 | 保存ボタン無効 or 「名前を入力してください」 |
| 名前が長すぎる（>100文字） | クライアントで切り詰め/警告。サーバ側でも 400 を返す |
| ソングブック数が上限（50件） | 「上限に達しました。不要なものを削除してください」 |
| 1ソングブックのスケール数が上限（200件） | 保存前に警告。サーバ側でも検証 |

### 2.3 読み込みの破壊的操作（最重要）

```
読み込みフロー（安全側）:
  1. 確認ダイアログ（「現在のソングファイルは上書きされます」）
  2. OK → API から取得（この時点ではまだ localStorage を書き換えない）
  3. 取得成功を確認してから localStorage に展開・タブ切替
  4. 取得失敗 → 現在のソングファイルは無傷・トースト表示
```

→ **取得成功を確定するまで現在データを消さない**（途中失敗でデータ喪失しない）。

### 2.4 サーバ側（Pages Functions）

| ケース | 返却 |
|--------|------|
| JWT 検証失敗 | 401 `{ "error": "unauthorized" }` |
| `user_id` 不一致の id 操作 | 403 |
| 必須フィールド欠落 / JSON 不正 | 400 `{ "error": "invalid_body" }` |
| `scales` が JSON.parse 不能 | 400 |
| D1 クエリ失敗 | 500 `{ "error": "internal" }`（詳細はログのみ、レスポンスに出さない） |
| 存在しない id | 404 |

## 3. 度数カラー（一括反映）

| ケース | 挙動 |
|--------|------|
| 「一括反映」を確認なしで誤爆 | 必ず確認ダイアログ（「全スケール N 件の個別設定を上書きします」） |
| ソングファイルが0件で一括反映 | ボタン無効 or 「対象スケールがありません」 |
| 不正なカラー値（JSON 破損データ） | persist のサニタイズで `DEFAULT_COLORS` にフォールバック |
| 個別設定中に対象スケールが削除された | 編集モーダルを閉じ、操作を破棄 |

---

## 4. ポジション表示/非表示

| ケース | 挙動 |
|--------|------|
| 全ポジション非表示 → 印刷で空指板 | 警告は出すが許容（ユーザー意図）。リセットで復帰可 |
| `visiblePositions` に不正キー（破損） | persist 読み込み時に `/^g\d+s\d+$/` でフィルタし不正キーを破棄 |
| `visiblePositions` が配列でも null でもない不正値 | `null`（全表示・既定）にフォールバック |
| 楽器切替でフレット数が変わった | 範囲外キーは描画時に無視（落とさない） |
| シリアライズ（localStorage/D1 は配列 or null で持つ） | 保存時 `visiblePositions ? Array.from(set) : null`、読込時 `Array.isArray(v) ? new Set(v) : null` |

## 5. リリースノート

| ケース | 挙動 |
|--------|------|
| `release-notes.json` の fetch 失敗 | モーダル内に「読み込めませんでした」。アプリは継続 |
| JSON パース失敗 | 同上 + `console.error` |
| `releases` 空/欠落 | 「リリースノートはまだありません」 |
| `highlights` 欠落・非配列 | 見出しのみ表示（クラッシュしない） |

## 5.5 曲共有（→ [SHARE.md](./SHARE.md)）

| ケース | 挙動 |
|--------|------|
| 作成時に未ログイン | 共有ボタンでログインを促す。ローカル機能は継続 |
| 作成 API 失敗（POST） | トースト「共有の作成に失敗しました」。リトライ可 |
| 作成上限（既定100件）到達 | 400 → 「共有の上限に達しました。古い共有が失効するまでお待ちください」 |
| スケール数が上限（200）超過 | クライアントで警告＋サーバ側でも 400 |
| 受け取り: 不正/失効した share_id | 404 → 「この共有は存在しないか、有効期限が切れています」 |
| 受け取り: 取得成功前にローカルを消さない | GET 成功を確定してから localStorage 展開（途中失敗でデータ喪失しない） |
| `?share=` が不正形式 | 無視して URL パラメータを除去し通常起動 |
| 期限切れ行が GET された | `expires_at <= now` は即 404（バッチ削除の遅延に依存しない） |
| share_id の UNIQUE 衝突 | サーバ側で再生成リトライ（露出しない） |
| 取り消し（DELETE）失敗 | 楽観的に消した行を一覧に復元 → トースト「取り消しに失敗しました」 |
| 取り消し対象が既に失効/他人のもの | 404 → 一覧から除去して再取得（実害なし） |
| オフライン | 送信/取得前に「オフラインです」を表示し API を叩かない |

## 6. localStorage 全般（既存方針の再確認）

| ケース | 挙動 |
|--------|------|
| 容量超過（QuotaExceededError） | トースト「保存容量が不足しています」。インメモリ状態は維持 |
| パース失敗（破損データ） | マイグレーション/サニタイズで既定値にフォールバック（既存 persist.js 方針） |
| プライベートモードで localStorage 不可 | 例外を握って機能継続（保存はスキップ、警告のみ） |

---

## 7. テスト方針

- **pure 層（DOM非依存）**: サニタイズ/バリデーション/フォールバックは Vitest で網羅
  - 不正 JSON → 既定値、上限超過 → 拒否、Set ⇄ Array 変換の往復、不正キー除去
- **API 層**: Pages Functions は 401/403/400/404/500 の分岐をテスト（D1 はモック）
- **UI/ネットワーク失敗**: 楽観的 UI のロールバックは手動/E2E で確認
