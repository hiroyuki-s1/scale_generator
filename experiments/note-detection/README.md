# 実験 01: ノート検出（単音ピッチ検出）

## 目的

マイクで弾いたギター/ベースの**単音**を、ブラウザ内で**リアルタイムに音名へ変換**できるかを
**実機で検証**する。製品の「練習モード（弾く→指板に反映→正誤＆スコア）」を仕様化する**前の
地固め**。headless では精度・遅延を測れないため、まず動く最小環境を作って自分の機材で体感する。

## 検証したいこと（成功条件）

| 項目 | 目標 / 見たいこと |
|------|------------------|
| **遅延** | 発音 → 音名表示まで **体感 80ms 以下**（低音弦ほど不利。実測する） |
| **精度（単音）** | クリーンな単音で**安定して正しい音名**。オクターブ誤検出の頻度 |
| **低音 vs 高音** | 6弦E(82Hz)〜1弦高域での検出安定性の差 |
| **電気 vs アコギ** | ライン入力（I/F）と内蔵マイク＋部屋鳴りでの精度差 |
| **頑健性** | 無音/減衰/ピックノイズで暴れないか（ゲート・しきい値の効き） |

→ これらが許容なら製品仕様（`docs/features/PRACTICE_MODE.md` 等）へ進む。ダメなら方式変更
（ML系 SwiftF0 等）を検討。

## スコープ（この実験で"やらないこと"）

- **和音/ポリフォニックは対象外**（単音のみ）。スケールを1音ずつ弾く前提。
- **弦・フレットの特定はしない**: 音から分かるのは**ピッチ（音名・オクターブ）まで**。
  同じ音は複数ポジションに存在するため、音だけで「どの弦の何フレット」かは原理的に不明。
- UIの作り込み・デザインはしない（計測できる最低限の表示のみ）。

## 技術方式（ゼロ依存・ノービルド）

ブラウザ標準APIのみ。ライブラリもビルドも不要。

```
マイク → getUserMedia → AudioContext → AudioWorkletNode
            └ AudioWorkletProcessor 内で YIN（古典DSP）でピッチ推定
            └ 周波数 → 最寄り音名 + セント誤差 + 信頼度 を main へ postMessage
main: 音名 / 周波数 / セント / 信頼度 / 計測遅延 を画面表示
```

- **YIN**（または pYIN/MPM）= 自己相関系の単音ピッチ推定。実装は純関数で書ける。
- **AudioWorklet** = 専用オーディオスレッドで低遅延処理（128サンプル/ブロック）。
- 低音は数周期必要なので**解析窓を可変**（低音=長め）にして精度を確保。
- **オンセット検出＋音量ゲート**で立ち上がり/減衰の誤検出を抑える。
- **オクターブ誤検出対策**: 期待レンジ（楽器の音域）でのバリデーション。

## 動かし方（コード作成後の手順）

AudioWorklet と getUserMedia は**セキュアコンテキスト必須**。`localhost` は安全とみなされる。

```bash
cd experiments/note-detection
python3 -m http.server 8000
# ブラウザで http://localhost:8000 を開く → マイク許可 → 楽器を弾く
```

> `file://` 直開きは AudioWorklet/マイクが動かないので不可。必ずローカルサーバ経由。
> HTTPS 不要（localhost は例外的に secure context）。

## 計測の進め方

1. 開放弦を1本ずつ鳴らし、表示音名が合うか・遅延の体感を見る。
2. メジャースケールを1音ずつ昇降し、追従と誤り（特にオクターブ）を観察。
3. 低音弦/高音弦、電気/アコギ、静かな部屋/雑音ありで比較。
4. 結果を下の「Findings」に追記。
