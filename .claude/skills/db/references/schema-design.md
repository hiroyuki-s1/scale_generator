# Schema Design — 正規化・キー・命名・データ型

## 正規化 (Normalization)

冗長と更新異常を防ぐためにテーブル/列を組織化する。設計初期から効かせる。

- **1NF**: 各セルは単一値。繰り返し列 (`tag1, tag2, tag3`) や CSV を持たない。
- **2NF**: 複合主キーの一部だけに依存する列を分離。
- **3NF**: 非キー列が他の非キー列に依存（推移従属）しない。

> 目安: まず 3NF まで素直に正規化する。`JOIN が多すぎて遅い` を**計測で確認してから**
> 限定的に非正規化（キャッシュ列・集計テーブル・JSON スナップショット）を入れる。
> 非正規化を入れたら「どこで一貫性を保つか」をコメント/コードで明示する。

### 正規化 vs JSON 列の判断
| 条件 | 推奨 |
|---|---|
| 個別フィールドで検索/結合/集計したい | 正規化（列に分ける） |
| 中身を DB で問い合わせず「丸ごと保存/取得」する | JSON 列 1 つで可 |
| スキーマが頻繁に変わる半構造データ | JSON 列（ただし検証はアプリで） |

このプロジェクトの `songbooks.scales` は「ソングファイルを丸ごとスナップショット」する
ので JSON (TEXT) が妥当。逆に「特定スケールだけ検索」要件が出たら正規化を検討。

## キー (Keys)

- **主キーは必ず置く。** 自然キーの組合せ（氏名＋生年月日 等）に頼らない → 重複事故の温床。
- **サロゲートキー**を基本にする:
  - SQLite/D1: `INTEGER PRIMARY KEY`（rowid エイリアスで最速）。`AUTOINCREMENT` は
    「再利用されない単調増加 ID」が要るときだけ（無いほうが速く軽い）。
  - 分散生成や URL 露出を避けたいなら UUID/ULID を TEXT で。
- **自然キー**には別途 `UNIQUE` 制約を貼り、人間が使う一意性を守る。
- **外部キー**は `<参照先単数>_id`（例 `user_id`, `songbook_id`）。FK 制約を宣言する。

## 命名規約 (Naming)

一貫した命名は「スキーマの文法」。新メンバーの理解とセキュリティルール実装を簡単にする。

| 対象 | 規約 | 例 |
|---|---|---|
| テーブル | 複数形・snake_case | `songbooks`, `user_settings` |
| 主キー | `id` | `id` |
| 外部キー | `<entity>_id` | `user_id` |
| 真偽 | `is_` / `has_` 接頭辞 | `is_public` |
| 時刻 | `created_at` / `updated_at` | （D1 は Unix ms INTEGER 運用） |
| インデックス | `idx_<table>_<cols>` | `idx_songbooks_user_id` |

## データ型 (Data Types)

- 意味に合う型を選ぶ。曖昧な型は索引効率もデータ整合も落とす。
- **時刻**: ISO8601 文字列 か Unix epoch 整数。プロジェクト内で統一する
  （このプロジェクトは `INTEGER`（Unix ms）で統一）。
- **真偽**: SQLite に BOOLEAN は無い → `INTEGER` 0/1 + CHECK。
- **金額/通貨**: 浮動小数を避け、最小単位の整数（例: 銭/cent）か DECIMAL 相当で。
- **列挙 (enum)**: TEXT + `CHECK (col IN (...))`、または別マスタテーブル + FK。
- **境界の検証**: 文字数・値域はアプリ検証に加え、DB の CHECK でも守る（外部入力は常に疑う）。

## 制約 (Constraints) を DB に置く

「DB が守れる整合性は DB に守らせる」。アプリのバグや別経路の書き込みから守れる。

- `NOT NULL` を既定に。NULL は本当に「未知/未設定」のときだけ。
- `UNIQUE` で一意性、`CHECK` で値域、`FOREIGN KEY` で関連。
- 削除伝播は意図して選ぶ: `ON DELETE CASCADE`（子も消す）/ `RESTRICT`（親削除を拒否）/
  `SET NULL`（参照を外す）。
